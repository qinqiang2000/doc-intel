# ApiAnything UI 设计文档

> 版本：v2.0 | 日期：2026-04-03
> 技术栈：Next.js 14 (App Router) + TailwindCSS + shadcn/ui

---

## 一、设计哲学

**单页核心，弹窗辅助**：整个产品以「AI 识别预览」为唯一核心页面，上传文档、矫正字段、调整 API 格式、调试优化全部在同一页面内完成。模板选择、API 生成等辅助流程通过弹窗承载，用户无需跳转页面即可完成从上传到生成 API 的完整旅程。

**字段驱动，逆向定位**：不使用传统的「框选图像区域 → 提取内容」，而是「AI 提取字段内容 → 逆向定位到文档区域高亮」。用户关注的始终是结构化字段，文档预览只是辅助验证手段。

**Prompt 泛化，不存位置**：用户画框或移动识别框时触发的专项 OCR，其结果用于丰富 Prompt 和自动调优，系统只保留 Prompt 方法，不保留「通过坐标识别特定内容」的硬编码逻辑，确保识别能力可泛化到任意同类文档。

```
┌─────────────────────────────────────────────────────────────────────┐
│  ┌──────┐                                          [用量] [头像▾]  │
│  │ Logo │  工作台  API管理  设置                                     │
│  └──────┘                                                          │

导航极简化，只保留三个入口：工作台（核心）、API 管理（查看已生成的 API）、设置。
模板市场、生成 API、沙箱调试等功能全部内嵌到工作台页面中，通过按钮+弹窗触达。

**路由结构**：

| 路径 | 页面 | 说明 |
|------|------|------|
| `/login` | 登录 | 邮箱+密码 / OAuth |
| `/register` | 注册 | 邮箱注册 + 组织创建 |
| `/workspace` | 工作台（核心页面） | 三栏布局，承载全部核心功能 |
| `/apis` | API 管理 | 已生成 API 的列表 + 详情 |
| `/apis/:id` | API 详情 | 调用统计 / Schema / 日志 / 文档 |
| `/settings` | 设置 | 账户 / 密钥 / 用量 / 团队 |

### 3.2 三栏详细设计

#### A 栏：文档预览区（55%）

**核心职责**：渲染文档原文 + 字段高亮叠加层 + 画框工具。

**文档渲染**：
- PDF：使用 PDF.js 渲染，支持缩放/翻页
- 图片（PNG/JPG）：直接渲染，支持缩放/平移
- Excel：表格预览（转 HTML 表格）

**字段高亮叠加层**（核心交互）：
- 在文档图像上方叠加一个 Canvas / SVG 图层
- 当用户在 B 栏 hover 某个字段时，A 栏自动高亮该字段对应的文档区域
- 高亮方式：半透明彩色矩形 + 字段名标签（如 `发票号码`、`行1:品名`）
- 不同字段组用不同颜色（基本信息=蓝色、货物明细=绿色、税额=橙色）
- 高亮区域通过「逆向定位」确定（见 3.4 节）

**工具栏**：
- 🔍+ / 🔍-：缩放
- ✋ 拖拽平移（默认模式）
- 📐 画框工具：用户在文档上框选区域 → 触发专项 OCR → 结果送入 B 栏（见 3.5 节）

#### B 栏：字段定义与矫正区（30%）

**核心职责**：展示和编辑 AI 提取的结构化字段，支持精细化的嵌套数据结构。

**字段展示结构**：采用分组卡片式设计，每个字段组是一个可折叠卡片。

```

**字段编辑能力**：

| 操作 | 方式 | 说明 |
|------|------|------|
| 修改值 | 直接点击字段值编辑 | 实时同步到 C 栏 JSON |
| 修改类型 | 点击类型标签切换 | string / number / date / boolean / array / object |
| 添加字段 | [+字段] 按钮或自然语言 | 支持手动和 AI 对话两种方式 |
| 删除字段 | 右滑或右键菜单 | 需确认 |
| 重命名 | 双击字段名 | 英文 key + 中文 label |
| 调整嵌套 | 拖拽或自然语言 | 字段提升/下沉到不同层级 |
| 数组管理 | [+行] 展开/折叠 | 添加/删除数组行 |
| 设置校验规则 | 校验规则卡片 | 数学关系、枚举约束、格式约束 |

**字段 ↔ 文档联动**：
- **hover 联动**：鼠标悬停 B 栏任一字段 → A 栏高亮对应区域
- **点击聚焦**：点击 B 栏字段 → A 栏自动滚动并放大到对应区域
- **数组行联动**：hover 货物明细的"行1" → A 栏高亮文档中整行货物信息

#### C 栏：API 结果预览（15%）

**核心职责**：实时展示 B 栏字段组合后的 API 输出 JSON + 验证状态 + 代码示例。

**内容**：
- API endpoint 信息（`POST /extract/{api_code}`）
- 实时 JSON 预览（随 B 栏编辑同步更新）
- Schema 验证状态（字段匹配、类型校验、业务规则校验）
- 多语言代码示例（Python / cURL / Node.js / Java）
- 历史版本切换（v1 / v2 / v3...）

**C 栏 ↔ B 栏联动**：
- hover C 栏 JSON 某个 key → B 栏对应字段高亮 + A 栏区域高亮
- C 栏 JSON 出现验证错误 → B 栏对应字段标红

### 3.4 逆向定位机制（字段 → 文档区域）

这是本系统区别于传统 OCR 工具的核心设计。

**原则**：不通过「在图上画框 → 提取框内内容」来定义字段，而是 AI 提取结构化内容后，逆向推断每个字段值在文档中的大致位置，用于高亮展示。

**实现流程**：

```
1. AI 处理器返回结构化 JSON
   → 同时返回每个字段的 location_hints（可选）

2. Schema Generator 补充位置信息
   → 对文档进行 OCR 获取全文 + 每个文字块的坐标
   → 用字段值在全文中做模糊匹配（fuzzy matching）
   → 匹配成功 → 记录 bounding_box {page, x, y, w, h}
   → 匹配失败（计算值等）→ 标记为 "derived"，不显示高亮

3. 前端渲染高亮
   → 在文档预览 Canvas 上叠加 SVG 矩形
   → 矩形颜色 = 字段组颜色，透明度 30%
   → 显示标签：字段中文名
```

**哪些字段有高亮，哪些没有**：

| 字段类型 | 高亮行为 | 说明 |
|---------|---------|------|
| 直接提取值（发票号、日期、名称） | ✅ 精确高亮 | 文档中有原文 |
| 数组行中的值（品名、数量、单价） | ✅ 行级高亮 | 高亮整行区域 |
| 计算值（合计、税额小计） | ⚠️ 最佳匹配 | 尝试匹配文档中的数字 |
| 衍生值（校验结果、聚合值） | ❌ 无高亮 | 标记 "derived" |

**位置信息不持久化到 Prompt**：位置坐标仅在当前会话的前端使用，不存入 PromptVersion，不影响 Prompt 泛化能力。

### 3.5 画框工具与专项 OCR（Prompt 自优化机制）

当 AI 自动提取的结果不准确时，用户可使用画框工具辅助矫正。

**交互流程**：

```
1. 用户点击 A 栏工具栏的 [📐 画框] 按钮，进入画框模式

2. 用户在文档上拖拽画出一个矩形框
   → 框选区域高亮，出现浮窗：
   ┌──────────────────────────┐
   │ 识别到的内容:             │
   │ "零件A M10×50 10 100 ..." │
   │                          │
   │ 这是哪个字段？            │
   │ ○ 新增为新字段            │
   │ ○ 矫正现有字段: [选择 ▾]  │
   │ ○ 作为辅助信息提供给 AI   │
   │                          │
   │ [确认]         [取消]     │
   └──────────────────────────┘

3. 用户选择操作：

   a) "新增为新字段"
      → 触发 correction engine：
        - 将 OCR 内容发送给 LLM
        - LLM 推断字段名、类型、在 Schema 中的位置
        - 更新 Prompt（增加提取该字段的指令）
        - 创建新 PromptVersion
      → B 栏新增字段，C 栏 JSON 更新

   b) "矫正现有字段"
      → 用户选择要矫正的字段（如 "行1:品名"）
      → 触发 correction engine：
        - 将「用户框选内容」+「当前字段值」+「原 Prompt」发送给 LLM
        - LLM 生成改进后的 Prompt 提取指令
        - 使用新 Prompt 重新处理文档
        - 创建新 PromptVersion
      → B 栏字段值更新，版本号递增

   c) "作为辅助信息"
      → OCR 内容作为额外上下文附加到对话中
      → 用户可在对话框进一步描述
```

**Prompt 自优化 (Auto-Research) 机制**：

```
用户框选矫正触发后：

Loop (max 3 轮):
  1. 生成/修改 Prompt
  2. 用修改后的 Prompt 重新处理文档
  3. 对比结果与用户期望
  4. 若结果满意 → 保存 PromptVersion，退出循环
  5. 若结果仍不满意 → LLM 自我反思，调整 Prompt 策略，继续

最终保存：
  - 仅保存最终的 Prompt 文本（PromptVersion）
  - 不保存框选坐标、位置信息、OCR 原文等
  - 确保 Prompt 可泛化到同类型的其他文档
```

**关键设计决策**：
- ✅ 保留的：Prompt 文本、Schema 定义、字段校验规则
- ❌ 不保留的：框选坐标、文档特定的位置信息、OCR 中间结果
- 这确保了一个在「发票A」上调好的 API，能正确处理「发票B」「发票C」

### 3.6 嵌套数据结构设计（精细化字段）

B 栏的字段编辑器需要支持复杂的嵌套结构，核心场景：

#### 场景一：货物明细（多行数组）

```json
{
  "items": [
    {
      "line_no": 1,
      "name": "A零件",
      "spec": "M10×50",
      "quantity": 10,
      "unit_price": 100.00,
      "amount": 1000.00,
      "tax_amount": 130.00
    },
    {
      "line_no": 2,
      "name": "B零件",
      "spec": "M12×60",
      "quantity": 5,
      "unit_price": 160.00,
      "amount": 800.00,
      "tax_amount": 104.00
    }
  ]
}
```

B 栏展示：卡片内每行可折叠，展开后显示子字段。

#### 场景二：税额汇总（数组 + 合计行）

```json
{
  "tax_details": [
    { "tax_type": "增值税", "tax_rate": "13%", "tax_amount": 234.00 }
  ],
  "tax_total": 234.00   // 🔗 = SUM(tax_details[*].tax_amount)
}
```

B 栏展示：数组下方有「合计」行，显示公式引用关系（🔗 图标）。

#### 场景三：数据逻辑加工规则

某些字段不是直接 OCR 提取，而是需要基于提取值进行逻辑计算。B 栏支持定义：

```
┌─ 📦 数据校验与加工规则 ──────────────────────┐
│                                              │
│  规则 1: amount = quantity × unit_price       │
│  规则 2: tax_total = SUM(tax_details.amount)  │
│  规则 3: grand_total = SUM(items.amount)      │
│                                    + tax_total│
│  规则 4: tax_rate 枚举: [3%, 6%, 9%, 13%]     │
│                                              │
│  容差: ±0.01 (浮点精度)                       │
│                                    [+ 添加规则]│
└──────────────────────────────────────────────┘
```

这些规则会：
1. 在 C 栏 API 结果中实时校验（显示 ✅ / ⚠️）
2. 写入 Prompt，让 AI 在提取时也做校验
3. 写入 ApiDefinition，公有云调用时自动校验

## 七、关键交互状态

### 文档处理状态机

```
上传中 (uploading) → 排队中 (queued) → 处理中 (processing) → 完成 (completed)
                                                              → 失败 (failed)
```

| 状态 | UI 表现 |
|------|---------|
| uploading | A 栏显示上传进度条 |
| queued | A 栏脉冲动画 + "排队中..." |
| processing | A 栏旋转 spinner + "AI 正在分析..." |
| completed | A 栏文档渲染 + B 栏字段填充 + C 栏 JSON 生成 |
| failed | A 栏错误提示 + 重试按钮 |

### 画框 OCR 交互状态

```
空闲 → [点击画框工具] → 画框模式
  → [拖拽画框] → 框选完成 → OCR 处理中（框闪烁）
  → OCR 完成 → 浮窗选择操作
  → [确认] → Prompt 优化中（B 栏骨架屏）→ 更新完成
```

### 矫正对话交互

- 用户输入 → B 栏进入 "更新中" 骨架屏状态
- AI 回复使用 SSE 流式输出，逐字显示在对话区
- Schema 变更时 B 栏使用 diff 动画（旧值淡出红色、新值淡入绿色）
- 版本号递增显示在底部会话历史条（如 `📄 发票A v3`）

### 三栏联动实现

```
workspace-store (Zustand)
  ├── activeFieldId: string | null     // 当前聚焦的字段
  ├── hoveredFieldId: string | null    // 当前 hover 的字段
  ├── fields: FieldGroup[]             // 全部字段定义（树形）
  ├── highlights: FieldHighlight[]     // 字段 → 文档位置映射
  └── jsonOutput: object               // 实时 JSON 输出

当 hoveredFieldId 变化时:
  → A 栏: 从 highlights 查找对应 bbox → 高亮 SVG 矩形
  → B 栏: 对应字段行高亮背景色
  → C 栏: 对应 JSON key 高亮
```
